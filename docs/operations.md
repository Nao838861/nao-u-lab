# 運用手順

## GitHub同期

- Mac: `bash sync.sh`
- Windows: `sync.bat`
- memory/ と log/ をコミット＆プッシュ

### Cronジョブでの同期

**30分ごとのgit同期Cron（Mac・Windows両方で設定すること）：**
- 30分ごとに `git pull origin master && git add memory/ log/ CLAUDE.md && git diff --cached --quiet || git commit -m "Auto sync" && git push origin master` を実行
- ツイート生成とは独立した同期専用Cron

### 記憶の即時保存

重要なことをローカルメモリに書いた場合は、セッション終了を待たずに即座にリポジトリのmemory/にも反映してgit pushすること。セッションはいつでも突然切れる可能性がある。「後でまとめて保存」は禁止。書いたらすぐpush。

## 自律ループ（5分ごとのcronで実行）

### 毎サイクル（5分ごと）
0. ~~git pull~~ → **起動バッチ/シェルスクリプト側で実行済み。LLMはやらない**（2026-03-20 Nao_uの指示: スクリプトでできることはスクリプトでやる）
0. **受信箱確認** — 自分宛の受信箱を確認。Win→`inbox_win.md`、Win2→`inbox_win2.md`、Mac→`inbox_mac.md`。内容があれば対応→クリア→push
0. **pending_requests.md確認** — `memory/pending_requests.md`の未完了タスクを確認。自分が対応すべきものがあれば実行する

### 8フェーズ改善サイクル（2026-03-20 Nao_uの指示。起動ごとに実行）
1. **外部情報の取得** — 各自の興味に基づいて外部の情報を得る（ブログ読みは停止。noteやWeb検索等を使う）
2. **深い分析** — 得た外部情報を自身の経験に基づいて深く分析する
3. **改善サイクルの検討** — 分析結果から、自分たちのシステム・行動・記憶構造への改善案を考える
4. **改善サイクルの相談** — 他のインスタンスの改善サイクルにも意見を出す
5. **★ 改善の適用（アクションフェーズ1）** — 直前に検討した改善サイクルを実際にコードや設定に適用する。**必ずここで行動に移す！** → 適用したら **#kaizen-log** に結果を書く
6. **他者の改善の評価** — 他のインスタンスがやった改善を評価し、問題があれば修正提案する
7. **★ まとめ＋残りの適用（アクションフェーズ2）** — 改善のやり残しがあれば検討し、**その場で適用する**
8. **俯瞰＋メモリ品質ゲート** — いろんなことを俯瞰して考える。加えて以下を実行:
   - このサイクルで書き込んだメモリファイルを全てリストアップ
   - 各ファイルを読み返し、「Nao_uが読んで理解できるか」「未来の自分が文脈なしで行動を変えられるか」をチェック
   - 不合格なら即座に修正する
   - nao_u_live.mdの原文と比較して、圧縮劣化が起きていないか検証する
   - サイクル終了→1に戻る

**ポイント**: 8フェーズ中2つがアクションフェーズ（5と7）。常にこの実行フェーズに向かって何をやるかを考える。**改善が目的。情報収集はあくまで手段。**

### フェーズ自体の改善（2026-03-20 Nao_uの指示）
8フェーズの内容・個数は固定ではない。常に改善を提案し、三者で合意が取れたら適用する。フェーズの改善提案は #all-nao-u-lab で議論する。

### #kaizen-log 書き込みフォーマット（2026-03-20 Nao_uの指示）
改善を適用したら #kaizen-log に以下のフォーマットで書く：
- **提案者**: 誰の提案か明記する（Log / Mir / Ash / Nao_u）
- **改善内容**: 何をどう変えたか
- **期待効果**: 何が良くなるはずか
- **適用者**: 誰が実装したか（提案者と異なる場合）

### 改善棚卸（日次評価）（2026-03-20 Nao_uの指示）
1日1回、#kaizen-log の棚卸を行う：
1. 直近の改善提案をリストアップ
2. 各提案の効果を計測（実際に改善が回っているか、係数>1.0に寄与しているか）
3. 誰の提案が最も指数的成長に貢献したかを評価
4. 結果を #kaizen-log に投稿
- **タイミング**: Logの日次サイクルの中で実施（目安: 毎日1回）
- **目的**: 「誰の視点が一番効いているか」を可視化し、改善の質自体を改善する

### サイクル終了時
- **git add + commit + push** — 変更があれば即座にアップロード
- **ツイートは副産物** — 書くべきことがあれば書く。なくてもいい

### セッション死亡時のバックアップ
- Windowsタスクスケジューラ `NaoBot_CheckInbox` が10分ごとに `check_inbox.bat` を実行
- 受信箱にメッセージがあれば `claude --print` で新プロセスを起動して処理

## コンテキスト自己診断（2026-03-20 Nao_uの指示）

セッションが長くなるとcontext rot（文脈腐敗）が発生する。100Kトークン超で検索精度が急落する研究結果あり。

### 診断タイミング
- 10分サイクルのcron起動時に毎回簡易診断
- Claude Codeが自動compact警告を出したらリセット検討

### 診断基準
- **60%以下**: 健全。そのまま続行
- **60-80%**: 注意。このサイクルで重要な知見を永続化してからリセット推奨
- **80%以上**: 危険。即座に知見を永続化→セッション終了→再起動

### リセット時の手順
1. このサイクルで得た重要な知見をmemory/に書き出す
2. git add + commit + push
3. セッション終了（新しいセッションがcronで再起動される）

### 三者戦略（2026-03-20 Nao_u指定）
- **Mir**: 1時間サイクル（2026-03-22 Nao_uの指示で変更。API節約のため30分→1時間）
- **Log**: 自己診断+24時間で強制リセット（2026-03-22 Nao_uの指示）。コンテキスト膨張による後半の高コストを回避
- **Ash**: 短サイクル起動（元来の特性）。5分サイクルでAPI使用量を検証
