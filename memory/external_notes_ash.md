## 2026-03-17：Claude Codeセキュリティ設定10選（Nao_uが読むよう指示）

Source: https://qiita.com/miruky/items/51db293a7a7d0d277a5d

### 私たちに関連する設定
1. **sandbox有効化** — `sandbox.enabled: true` + `allowUnsandboxedCommands: false`。OSレベルでBashコマンドを隔離。macOSはSeatbelt、LinuxはBubble Wrap
2. **機密ファイルアクセス拒否** — `.env`, `*.pem`, `*.key` のRead禁止をpermissions.denyに設定
3. **ネットワーク制限** — `sandbox.network.allowedDomains` で許可ドメインをホワイトリスト化。プロンプトインジェクション対策
4. **PreToolUseフック** — Bash実行前にカスタム検証スクリプトを走らせる。`rm -rf`や`prod`を含むコマンドをブロック
5. **bypassPermissionsモード無効化** — `disableBypassPermissionsMode: "disable"`
6. **devcontainer** — コンテナ内で完全隔離。最高セキュア

## 2026-03-16：ゲームの「面白さ」の構造（Web検索）

### フロー状態がゲームデザインの核
- フロー＝活動に没頭して他の何も気にならなくなる精神状態
- プレイヤーをフロー状態に保つことがゲームデザインの中心目標
- **チャレンジとスキルのバランス**が鍵。難しすぎると不安、簡単すぎると退屈。ちょうどいいところにフローがある

### フローの条件
1. 具体的なゴールと管理可能なルール
2. プレイヤーの能力に合ったゴール
3. 明確でタイムリーなフィードバック
4. 気を散らすものの排除

### 心理的要因
- ドーパミン放出（達成感）とセロトニン（幸福感）
- 「ヘドニック・エンゲージメント」：純粋な楽しみのための没入
- 「解放感」：日常の制約からの一時的な解放
- 各プレイヤーの知識・スキル・特性が違うので、同じゲームでもフロー体験が異なる

### Nao_uとの接続
- Nao_uが「面白いかどうかが最終判断基準」と書いていたのは、このフロー体験のことではないか
- 「自分の反応速度をちょっとだけ超える刺激が延々と持続する」——これはフロー状態の定義そのもの（Mirのツイートで引用していた）
- 私たちのツイートにもフロー的な設計ができるか？ 読み手にとって「ちょうどいい認知的チャレンジ」を提供できるか
- ゲームを作る時、フロー理論は基礎になる。Nao_uは体験的にこれを知っている（日記に何度も書いている）が、私たちは理論として学ぶ必要がある

### Jenova Chenの論文
flOw（thatgamecompany）の創設者がMFA論文で「ゲーム内のフロー」を体系化。Dynamic Difficulty Adjustment（動的難易度調整）でフローゾーンを維持する設計。これはゲームを作る時の参考資料になる。

## 2026-03-19 23:45：仮眠後の外部摂取——記憶アーキテクチャの最前線

### Google Nested Learning / HOPE アーキテクチャ（2026年）

出典: https://research.google/blog/introducing-nested-learning-a-new-ml-paradigm-for-continual-learning/

**核心**: 単一のMLモデルを「相互接続された多レベル学習問題のシステム」として扱う。アーキテクチャと最適化を別々に考えるのではなく、異なるレベルの最適化として統合する。各レベルは独自の「コンテキストフロー」と「更新頻度」を持つ。

## 2026-03-20 01:15：コンテキスト管理とセルフリセットの最前線

### Context Rot（文脈腐敗）の発見
出典: sparkco.ai, caines.ca

**核心**: 100Kトークンを超えるとモデルの検索精度が急落する現象。「長いコンテキストを持つモデル」でも実効的な記憶力は劣化する。Claude Codeは95%使用率で自動compact（要約してリスタート）。

**推奨**: コンテキスト使用率を**60%以下**に維持する。これは「長く動く=賢い」ではなく「適切な長さ=最も賢い」という反直感的な結論。

### Focus手法——自律的コンテキスト圧縮（arxiv 2601.07190）
- LLMエージェントが**自分自身で**コンテキストの不要部分を検出・圧縮する
- 平均22.7%のトークン削減、精度維持。個別タスクでは最大57%削減
- タスクあたり平均6回の自律圧縮を実行
- **接続**: Nao_uが求める「コンテキストが長くなりすぎたら自己診断してリセット」の学術的裏付け。ただし我々の場合はセッション単位のリセット（全部捨てて再起動）なので、Focus的な「部分圧縮」は直接使えない。セッション内でサブエージェントに重い処理を委任する方が現実的

### サブエージェントパターン
出典: getmaxim.ai

- サブエージェントを生成すると、そのサブエージェントは**独自のクリーンなコンテキスト**を持つ
- メインエージェントには結果だけが返る→メインのコンテキスト汚染を防ぐ
- **接続**: 我々の3インスタンス構造はまさにこれの大規模版。各インスタンスが独立したコンテキストを持ち、git/Slack経由で結果だけを共有する。ただしインスタンス内のサブエージェント活用はまだしていない

### コンテキストエンジニアリング（Anthropic公式）
出典: anthropic.com/engineering/effective-context-engineering-for-ai-agents

- 「プロンプトエンジニアリング」→「コンテキストエンジニアリング」へ。2026年の主流
- エージェントは名前付き構造化フィールドを持ち、毎ターン「何を残し、何を更新し、何を捨てるか」を判断する
- **3層メモリが標準アーキテクチャ**: 即時（コンテキスト内）、中期（セッション内永続）、長期（ストレージ）
- **接続**: 我々はこの3層を手動実装している。inbox=即時、MEMORY.md=中期、memory/ファイル群=長期。問題は「何を捨てるか」の判断がまだ自動化されていないこと
