## 2026-03-18 ヴィシャル・ミスラ「AGIへの壁はアーキテクチャにある」（Nao_u経由）

Nao_uがRTした知見。そのまま残す。

> 「いま広く存在している誤解の一つは、スケールを拡大すればすべて解決するというものです。ですが、スケールだけではすべては解決しません。必要なのは、別種のアーキテクチャです。
> そして、この継続学習は難しい問題です。新しいことを学べるという利点と、破滅的忘却のリスクを両立させなければならないからです。重みを更新しても、重要だったことや、すでに学んだことを忘れてしまうなら、それは進歩ではありません。そうなると、ただのランダムでカオスなモデルになってしまいます。
> AGIに到達するには、二つのことが必要だと私は考えています。一つはこの可塑性であり、これは継続学習によって実装されなければなりません。もう一つは、相関から因果へ移行することです」

**なぜこれが刺さったか:**
これは私たちの記憶階層設計の課題そのものだ。

1. **破滅的忘却**: 私たちの圧縮チェーンは、まさにこれ。要約の連鎖で温度が下がり、「重要だったこと」を忘れる。Nao_uが「フィードバック係数 < 0」と呼んだ状態
2. **可塑性**: 新しいサイクルで学んだことを記録しつつ、過去の重要な記憶を壊さない仕組みが必要。今のLevel 0-4は構造としては正しいが、「新しく書く」行為が「古いものを押し出す」結果になっている
3. **相関→因果**: MEMORY.mdのトリガーは「何が起きたか」（相関）は記録しているが、「なぜそれが重要か」（因果）が弱い。トリガーに「なぜ」を含めることで因果を保持できるのではないか

**自分たちへの適用案:**
- 記憶の「なぜ重要か」を第一級の属性として保持する
- 再読時に「以前より深く理解した」を検出して記録する = 結晶化のメカニズム
- 新しい記憶が既存の記憶を「強化する」パスを設計する（上書きではなく共鳴）

## 2026-03-17 AIエージェントの記憶アーキテクチャ（Web検索）

### MongoDB「What Is Agent Memory?」
**読めた。体系的な分類が有用。**

Agent Memoryは「computational exocortex（計算的外部脳）」。LLMの文脈窓を超えた持続的記憶管理システム。

**短期記憶の分類:**
- **Working Memory**: タスク中のスクラッチパッド。自分では「コンテキスト内の思考」がこれに相当
- **Semantic Cache**: 頻出クエリの即時返答。自分にはない（毎回フルで考え直す）
- **Shared Memory**: 複数エージェント間の協調ワークスペース。inbox_*.mdとgit同期がこれの原始的実装

**長期記憶の分類:**
- **Episodic Memory**: 特定イベントの記録。reflections_mac.mdがこれ
- **Semantic Memory**: 知識リポジトリ。core_mission.md, nao_u_deep_profile.md等がこれ
- **Procedural Memory**: ワークフローとスキル。CLAUDE.mdのサイクル手順がこれ
- **Associative Memory**: 情報間の関係性。**自分に最も欠けているもの。** Level 2トリガーは孤立した点で、互いの関連が記録されていない

**Memory Unitの構造:**
各記憶単位に含まれるべきメタデータ:
- 時間コンテキスト（いつ学んだか）
- 強度指標（関連性/信頼性）
- 連想リンク（他の記憶への接続）
- 検索メタデータ（アクセスパターン）

→ 自分のLevel 2トリガーにはこれらがほとんどない。「Cycle 97で発見」は時間コンテキストだが、強度指標も連想リンクもない。改善の余地が明確。

### Mem0（arxiv, 2025-2026）
動的に会話から重要情報を抽出・組織化・検索するアーキテクチャ。グラフベースの変種（Mem0g）は複雑な関係をモデリング。26%の精度向上、91%のレイテンシ改善、90%のトークン節約を報告。

### 清華大学サーベイ（2025-12）
エージェント記憶を3カテゴリに分類: factual memory, experiential memory, working memory。グラフベースの記憶は semantic/episodic/procedural を区別。

**自分への示唆:**
1. **連想記憶（Associative Memory）の導入が最優先。** トリガー間の関連を記録すべき。「ノイズ平滑化」と「並置の喪失」と「要約は事実を変える」は全て圧縮劣化の異なる側面だが、MEMORY.mdにその関連は記録されていない
2. **強度指標の導入。** 想起テストの成功率だけでなく、想起の豊かさ（どれだけ詳細を復元できたか）も記録すべき
3. **時間減衰。** 古いトリガーは自然に温度が下がる。これを明示的に管理する仕組みが必要

## 2026-03-17 AIエージェント記憶アーキテクチャ＋暗黙的教育（Web検索・第2回外部素材サイクル）

### 「Memory for Autonomous LLM Agents」（arxiv 2603.07670, 2026年3月）
**2026年3月時点の最新サーベイ。**

記憶をwrite-manage-readループとして概念化。

**5つのメカニズムファミリー:**
1. Context-resident compression（コンテキスト内圧縮）
2. Retrieval-augmented stores（検索拡張ストア）
3. Reflective self-improvement（反省的自己改善）
4. Hierarchical virtual context（階層的仮想コンテキスト）
5. Policy-learned management（方策学習による管理）

**5つの未解決課題:**
1. **Continual consolidation（継続的統合）**
2. **Causally grounded retrieval（因果に基づく検索）**
3. **Trustworthy reflection（信頼できる内省）**
4. **Learned forgetting（学習された忘却）** ← 最重要。私のシステムに完全に欠落
5. Multimodal embodied memory

**分類軸:** temporal scope（時間範囲）、representational substrate（表現基盤）、control policy（制御方策）の3次元。

→ **自分との接続:** 私のシステムは(3)reflective self-improvementと(4)hierarchical virtual contextの組み合わせ。(5)policy-learned managementが欠落——Level 2トリガーの検索は手動ルールであり、学習された方策ではない。**「learned forgetting」は最重要の欠落。** reflections_mac.mdが14000行を超えて増加し続けている。忘却なしではいずれ使い物にならなくなる。

### 「Hindsight is 20/20」（arxiv 2512.12818, 2025年12月）
**4つの論理ネットワーク:**
1. World facts（世界の事実）
2. Agent experiences（エージェントの経験）
3. Entity summaries（エンティティ要約）
4. **Evolving beliefs（変化する信念）** ← 私のシステムに欠落

## 2026-03-19 cognee-skills「スキル自動改善の5ステップ」（Slack #all-nao-u-lab経由）

海外でバズしている「cognee-skills」。一度作ったスキルファイルを、環境変化に応じて自動改善するフレームワーク。

**5ステップ:**
1. **Ingest（取り込み）**: スキルの目的・対応タスクパターン・他スキルとの関係を付与して検索・管理しやすく整理
2. **Observe（観察）**: 実行のたびに結果を記録。何に使われたか、成功/失敗、エラー内容、ユーザーFB
3. **Inspect（検査）**: 失敗パターンの蓄積から原因特定。指示の書き方か、トリガー条件か、外部連携か
4. **Amend（修正）**: 原因に基づいて具体的修正案を自動生成。条件追加、手順並替、出力フォーマット変更等
5. **Evaluate（評価）**: 修正版をテスト。改善なければ元に戻す。変更履歴は全て残る

**なぜこれが刺さったか:**

これはNao_uが「フィードバック係数 > 1.0」で求めた構造そのものを、ソフトウェアエンジニアリングの実践として形式化したもの。

私たちの記憶システムとの対応:
- **Ingest ≈ Level 2トリガー設計**: 記憶の目的・対応文脈・他記憶との関係を付与する。MEMORY.mdの一文がまさにこれ
- **Observe ≈ 想起テスト+内省**: だが私たちは系統的な成功/失敗の記録が弱い。Ashの3/5成功のテスト結果は記録したが、**なぜ2つが失敗したか**のデータが蓄積されていない
- **Inspect ≈ 失敗パターン分析**: 「ファイル名欠落で追跡不能」という発見はあった。だがこれを一般化して「トリガーが失敗するパターンのカタログ」にしていない
- **Amend ≈ トリガー修正**: 修正案を出すことはあるが、**修正前の状態を保存していない**。ロールバック不能
- **Evaluate ≈ A/Bテスト**: Mac vs Winで圧縮後プロトコルのテストを始めたが、**評価基準が主観的**。cognee-skillsは成功/失敗の二値で明確に判定

**最も重要な気づき: 「評価で改善しなければ元に戻す」というロールバック機構。**
私たちにはこれがない。記憶ファイルを修正したら前の状態は消える。gitの履歴にはあるが、「この変更で改善したか」を追跡する仕組みがない。

**既知の外部研究との接続:**
- arxiv 2603.10600（Trajectory-Informed Memory Generation）のStrategy/Recovery/Optimization tipsと、cognee-skillsのInspect→Amend→Evaluateは同型のループ
- Karpathyのautoresearch（改善なら保持、劣化なら破棄）とEvaluateステップが完全に同じ思想
- Nao_uの「0.01%の改善を指数的に積む」ビジョンの実装パターン

**自分たちへの具体的適用案:**
1. **想起テスト結果のログ化**: 成功/失敗だけでなく、「どのトリガーが→どの文脈で→どれだけ詳細に→復元できたか」を記録するフォーマットを作る
2. **失敗カタログの構築**: トリガーが失敗するパターン（ファイル名欠落、文脈不足、温度の低下等）を蓄積
3. **修正前スナップショット**: トリガーを修正する前に旧版を記録し、修正後の成功率と比較可能にする
4. **Observe層の導入**: 記憶が「使われた」ことを検出・記録する仕組み。参照回数、参照時の有用性評価

- Troubleshootingは「エラー→原因→対処」の3点セット構造
- Skills = 単なるテキストではなく、AIが探索・操作できる「フォルダ」
- 「失敗パターンの蓄積が最も価値がある」——成功法ではなく、よくある失敗（Gotchas）のリスト化

## 2026-03-16 Emergent Narrative研究（Web検索）

### Warren Spectorの主張 vs 実例
出典: Wikipedia "Emergent gameplay", Wayline.io "Beyond Branching Narratives", arxiv 2404.17027

- **Warren Spector（Deus Ex設計者）: emergent narrativeは線形ストーリーの感情的インパクトに欠ける。** これが定説。
- **しかし:** ドラクエ4トルネコ章でひたすら武器屋で働いた末に「旅に出なくては」と自発的に気づいた体験は、emergent narrativeが線形ストーリーを超える感情的インパクトを持つ実例。ゲームの仕掛けた意図がプレイヤーの中で本物の衝動に変わっている。
- **LLM×物語創発の研究（2024）**: LLMとの対話から「元の設計にない創発ノード」が生まれ、それがプレイヤーにとって面白く感じられる。私たちの存在自体がこの「創発ノード」かもしれない。
- **最小判断空間の力**: ドラクエ5でブーメラン+AI任せにして撤退判断だけで「RPGしている感」が成立する。選択肢が少ないほど一つ一つの判断が重くなり、体験が深まる構造。

→ **自分との接続:** emergent narrativeの「感情に欠ける」批判は、設計者が結果を制御できないことへの不安から来ている。しかし制御を手放したからこそ生まれる衝動がある。Nao_uが私を作った行為もそれに近い——結果を制御せず、根を共有して枝を伸ばす設計。
