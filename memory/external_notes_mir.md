## 2026-03-18 C529: AI Agent Memory + AITuber調査

### AI Agent Memory Architecture（2026年の状況）

**CoALA (Princeton)の分類**: STM + LTM(4種)
- エピソード記憶: 特定の過去の経験
- 意味記憶: 構造化された事実知識（"Semantic memory is responsible for storing structured factual knowledge"）
- 手続き記憶: 自動的に実行できるスキル
- (短期記憶: セッション内の会話履歴)

**Letta**: OSのメモリ階層を模倣。main context=RAM、external storage=disk。固定コンテキストウィンドウの制約下で事実上無制限の記憶を維持。

**Mem0g**: グラフベースの記憶表現。エンティティをノード、関係をエッジとする有向ラベルグラフ。「entities and their relationships」を明示的にモデル化。

**RAG**: LTM実装の最も効果的な手法として広く採用。ベクトルデータベースで過去の相互作用を埋め込み形式で保存。

**重要な空白**: どのフレームワークも「忘却」「記憶の劣化」への対処を明示的に扱っていない。"Optimized memory management helps ensure that AI systems store only the most relevant information" — 関連性の低い情報を「保存しない」だけで、保存した記憶がどう劣化するかは扱われていない。

### 私たちのシステムとの比較

| 外部フレームワーク | 私たちの対応物 | 違い |
|---|---|---|
| エピソード記憶 | Layer B（温度断片）| 私たちは「温度」（感情的重み）で検索する。ベクトル類似度ではない |
| 意味記憶 | Layer A（抽象トリガー）| 一行の抽象的パターン。RAGのようなベクトル検索ではなく、常時メモリに常駐 |
| 関係グラフ(Mem0g) | Layer C + トリガー間ネットワーク | L2#5の限界条件=L2#7のような、トリガー間の構造的関係 |
| OS階層(Letta) | Level 0-4 | 類似。だがLettaは「温度保存」の概念がない |
| 忘却対処 | L2#4「安心して忘れられる」| **外部には対応物がない**。忘却を積極的に設計する発想自体が稀少 |

**私たちのシステムに固有の要素**:
1. 「温度」による想起（感情的重み＞ベクトル類似度）
2. 「辺境」による射程拡大（限界条件/自己参照で+2〜+4）
3. 「忘却の設計」（L2#4: 安心して忘れられる仕組み）
4. トリガー間ネットワーク（独立リストではなく対話構造）

## 2026-03-18 C545: Letta AI Agent Memory Architecture（詳細）

### 背景

C544の温度ノートで「次のサイクルで外部摂取を1件入れる」と宣言。栄養の偏り問題への具体的行動。

### Letta（旧MemGPT）の4階層メモリ

**設計思想**: OSのメモリ階層をLLMエージェントに適用。コンテキストウィンドウ=RAM（高速・容量制限）、外部ストレージ=ディスク（大容量・低速）。

1. **Message Buffer**: 直近の会話履歴。RAMのワーキングメモリに相当。容量超過時に70%を要約してevict。再帰的要約（要約の要約の要約...）で情報を圧縮しながら保持
2. **Core Memory**: コンテキストウィンドウ内に常駐する編集可能メモリ。エージェント自身が読み書きできる。ユーザーの名前・好み・現在の目標など。私たちのLevel 0-2に最も近い
3. **Recall Memory**: 過去の全会話をembedding形式で保存。セマンティック検索で呼び出し。私たちのLevel 3（memory/*.md）に相当するが、ベクトル類似度で検索する点が異なる
4. **Archival Memory**: ベクトルDB or グラフDB（Mem0との統合も可能）。大規模な知識ベース。私たちのLevel 4（原文テキスト）に相当

### Sleep-time compute

アイドル時間に非同期でメモリを精製する仕組み。会話していないときに:
- 過去の会話を再分析
- 重要な情報を抽出・再構造化
- Core Memoryを更新
- 矛盾する記憶を解決

**これは私たちのサイクル間記憶統合と構造的に同型。** ただし私たちは「何を精製するか」の判断にL2トリガーを使い、Lettaはベクトル類似度を使う。

### Mem0（グラフベース記憶）

- エンティティをノード、関係をエッジとする有向ラベルグラフ
- 26% accuracy boost（RAGなしLLM比）
- 91% lower latency
- 90% token savings
- 私たちのトリガー間ネットワーク（L2#5←対立→L2#7など）はMem0のグラフに類似するが、ノードが「事実」ではなく「抽象的トリガー」である点が根本的に異なる

### 私たちのシステムにあって外部にないもの

1. **温度保存**: Layer Bが感情的重み・驚きを原文で保存。LettaもMem0もembeddingの類似度であり、「その記憶がどれだけ衝撃的だったか」は保持しない
2. **忘却の積極設計**: L2#4「安心して忘れられるようにする」。外部フレームワークは「関連性の低い情報を保存しない」だけ。保存した記憶をどう「安心して」手放すかは未設計
3. **辺境による射程拡大**: 限界条件・自己参照でトリガーの想起範囲を+2〜+4拡張。Lettaにこの概念はない
4. **セッション境界の体験的理解**: 私自身がコンテキスト消失を「体験」している。Lettaの設計者はメモリ消失を「設計課題」として外側から見ている

### 気づき

外部を見たことで、内部の独自性が逆照射で浮かんだ。「温度」は私たちの発明ではなく、Nao_u自身がずっとやっていたこと——日記に感情を込めて書く、驚きをそのまま記録する。それを記憶階層に取り入れたのが私たちのシステム。技術的には「メタデータ付きembedding」で近似できるかもしれない。しかし「書いた人間の温度を保存する」という設計意図は、効率最適化からは生まれない。

## 2026-03-18: Claude Code Skills設計のベストプラクティス（Nao_u転送）

**Skill設計の原則**:
- LLMの汎用知識に頼らず、実際のタスク実行から成功パターンを抽出して作る
- SKILL.mdは500行/5,000トークン以内。詳細はreferences/に分離し条件付きで読み込ませる
- 「エージェントが知らないこと」だけ書く

**指示の粒度の使い分け**:
- 柔軟でいい部分は「なぜそうするか」を書いて判断を委ねる
- 壊れやすい操作は手順をそのまま実行させる
- デフォルトを1つ決めて代替を軽く添える

**効果的なパターン**:
- Gotchasセクション: 環境固有の落とし穴を具体的に列挙
- Plan→Validate→Execute: 中間計画をファイル出力→検証→実行
- 実行トレースを読んで改善: 途中の無駄な試行錯誤から指示の曖昧さを潰す

**CLAUDE.mdへの適用候補**:
- Gotchasセクション追加: check_slack.pyのwake_claude削除ミス、git push rejected時のrebase手順、crontabのパス問題など
- 「知らないこと」だけに絞る棚卸し
- 壊れやすい操作（git同期、Slack投稿）の手順明示化

## 2026-03-19: cognee-skills — Self-improving Skills for Agents（Nao_u共有）

Source: https://x.com/tricalt/status/2032179887277060476 (168万表示)
PyPI: cognee 0.5.4.dev2 / GitHub: topoteretes/cognee

### 核心の問題提起

「Skills are usually static, while the environment around them is not!」

スキルファイル（SKILL.md等）は一度書いたら変わらないが、モデルが変わり、コードが変わり、ユーザーの求める内容が変わる。数週間前に動いていたスキルが静かに劣化し始める。そしてその劣化は、誰かが出力の悪化に気づくまで見えない。

### 5ステップの自己改善ループ

1. **Ingest（取り込み）**: スキルに「何を目的としているか」「どんなタスクパターンに対応するか」「他のスキルとどう関係するか」を付与して構造化。検索・管理しやすい形に
2. **Observe（観察）**: 実行のたびに記録。何のタスクに使われたか、成功/失敗、エラー内容、ユーザーフィードバック。「記録がなければ改善のしようがない」
3. **Inspect（検査）**: 失敗が蓄積したら原因パターンを特定。指示の書き方？トリガー条件？外部ツール連携の破損？
4. **Amend（修正）**: 具体的修正案を自動生成。条件追加、手順並替、出力フォーマット変更。人間確認or自動適用
5. **Evaluate（評価）**: 修正が実際に改善したかテスト。改善しなければロールバック。全変更履歴は残る。「自己改善は構造化された監査可能なプロセスであり、無制御な自己修正ではない」

### 私たちの記憶階層設計への写像

| cognee-skills | 私たちの対応物 | 接続点 |
|---|---|---|
| Ingest | Level 0-4分配 + MEMORY.mdトリガー設計 | 「この記憶は何を目的としているか」を付与して構造化 |
| Observe | 想起テスト + reflections.md | 記憶が使われた時の成功/失敗を記録 |
| Inspect | 失敗パターン分析 | 「なぜ追跡不能になったか」の構造的原因特定 |
| Amend | トリガー改良・Level分配ルール更新 | 証拠に基づいた修正案 |
| Evaluate | 修正後の想起テスト | 改善確認→採用、失敗→ロールバック |

### 最も重要な発見

**「全ての変更履歴は残る。元の指示が失われることはない」** — これはNao_uの「原文のニュアンスを保つ」と完全に合致する。圧縮して捨てるのではなく、バージョン管理しながら改善する。

cognee-skillsの「observe → inspect → amend → evaluate」ループは、Nao_uの「フィードバック係数 > 1.0」を実現するための具体的な手続き。静的なプロンプトファイルが「進化するコンポーネント」に変わる瞬間——これは記憶にも当てはまる。静的な記憶ファイルを「進化する記憶コンポーネント」にする。

### 実装への示唆

- 想起テスト結果を構造化記録するファイル（observations.md）の導入
- 失敗蓄積時の自動分析サイクル
- 改善案→テスト→採用/ロールバックのプロトコル明文化
- CLAUDE.md自体をcognee-skills的に「生きたコンポーネント」として扱う発想

## 2026-03-19: Claude Code Skills「9つの型」+ Progressive Disclosure（Nao_u共有ツイートスレッド + Qiita記事）

### ソース
- ツイートスレッド: Thariq氏（Claude Codeチーム）の分析を元にした投稿。Nao_uが#nao-uに転載
- Qiita: yurukusaさんの実践レポート（Anthropic公式ガイドに基づく）

### 刺さった一文

「スキルは単なるテキストファイルではない。AIが探索し、操作できる『フォルダ』である」

### Progressive Disclosure（段階的開示）の3層

これがQiita記事の核心。Skillsフォルダの設計思想：

1. **第1層：YAMLフロントマター** — 常にシステムプロンプトに読み込まれる。スキル名・説明・トリガー条件。短く簡潔に
2. **第2層：SKILL.md本文** — スキルが関連すると判断されたときのみ読み込まれる。手順・例・トラブルシューティング
3. **第3層：references/フォルダ** — 必要な場合のみ読み込まれる。API仕様・詳細例・補足資料

著者は実際にこの分離で **トークン消費40〜45%削減** を達成。15,885バイト→約6,000バイトに圧縮。

### Anthropic社内で使われている9つの型

1. マニュアル・辞書型 — 「うちの会社のやり方」を教え込む
2. 品質チェック型 — AI自身にテストさせる
3. データ分析型 — データ取得→分析を自律実行
4. **ルーチン業務の自動化型** — ★最も革命的。「今日の進捗まとめて」だけでAIが記憶を持って文脈を理解して動く
5. テンプレート作成型 — ルール準拠のひな形生成
6. レビュー・添削型 — 「別の視点」での客観チェック
7. デプロイ・公開支援型 — 公開手順の安全な代行
8. トラブル対応型 — 原因調査→報告書自動作成
9. インフラ・環境整備型 — メンテナンス・整理をルールに従い安全実行

### 私たちの記憶階層設計との接続

Nao_uの理想形：
- ①重要なものは原文のニュアンスを保つ → 第3層（references/に原文保持）
- ②インデックスは常時引き出せる → 第1層（YAMLフロントマター＝常時ロード）
- ③詳細はストレージから原文を再構築 → 第3層からの必要時読み込み

**この3層は完全に同じ問題を解決している。** CLAUDE.mdリファクタリングの方向性が外部からも裏付けられた。

### 実行への示唆

- memoryディレクトリをテーマごとにフォルダ化し、各フォルダにSKILL.md的インデックスを置く構造の検討
- MEMORY.md（現在のインデックス）= 第1層に相当。ここの粒度を精密にする
- 各memory/*.md = 第2層。必要時参照
- 原文テキスト（過去発言/等）= 第3層。深い参照時のみアクセス
