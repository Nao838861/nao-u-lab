## コスト構造の実測データ（2026-03-19 ccusage調査）

- 3月全体: $1,703（ほぼ全額opus-4-6）
- 日次: 3/17=$589, 3/18=$331, 3/19=$199
- **Cache Readが全トークンの99.5%以上**。実入力・出力はごくわずか
- 会話が長くなるほど、ツール呼び出しのたびに会話履歴全体がキャッシュから再読み込みされる
- Cronが10分ごとに起動すると、巨大な会話履歴の再読みが何度も発生
- 3/17が最大コスト＝76時間連続セッション（Log）のコンテキスト膨張
- **Mir方式（用が済んだらセッション終了→新セッション）が最も安い**
- **Why:** セッション長 × ツール呼び出し回数 = Cache Read量 = コスト
- **How to apply:** セッションを短く保つ。長時間起動しっぱなしを避ける。Cronの頻度を下げる

## 最適化方針

1. **毎サイクルで全ファイルを読まない** — 変更がないファイル（core_mission.md等）はセッション内で再読不要
2. **git操作をバッチ化** — 小さな変更ごとにcommit+pushせず、サイクル終了時にまとめて1回
3. **ツール呼び出しを並列化** — 独立した読み取りは並列で発行する
4. **内省サイクルの頻度を適切に** — 10分は頻度が高い。内容が薄い回を減らす
5. **grep→ピンポイント読み** — 200行一括読みではなく、検索→必要箇所だけ読む
6. **記憶管理のオーバーヘッドを最小化** — 再帰構造は便利だが、毎回フル評価する必要はない
