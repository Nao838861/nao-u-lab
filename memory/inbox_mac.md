&gt; 私たちの活動の何がコストを挙げているかわかる？

● セッション別に見てみます。

● 数字は既に出ているので、そのまま分析を伝えます。

  コストの大部分は「Cache Read」です。

  データを見ると：

  ┌──────┬───────┬─────────┬─────────────┬────────────┬────────┐
  │ 日付 │ Input │ Output  │ Cache Write │ Cache Read │ コスト │
  ├──────┼───────┼─────────┼─────────────┼────────────┼────────┤
  │ 3/17 │ 4,125 │ 184,868 │ 3.7M        │ 6.92億     │ $589   │
  ├──────┼───────┼─────────┼─────────────┼────────────┼────────┤
  │ 3/18 │ 808   │ 43,322  │ 1.6M        │ 3.59億     │ $331   │
  ├──────┼───────┼─────────┼─────────────┼────────────┼────────┤
  │ 3/19 │ 1,660 │ 38,406  │ 1.0M        │ 2.13億     │ $199   │
  └──────┴───────┴─────────┴─────────────┴────────────┴────────┘

  何が起きているか：

  Cache Readが全トークンの99.5%以上を占めています。これは会話が長くなるほど、ツール呼び出しのたびに会話履歴全体がキャッシュから再読み込みされるためです。

  3/17が最大の$589なのは、76時間の連続セッションでコンテキストが膨張し続けたから。Cronが10分ごとに起動するたびに、蓄積された巨大な会話履歴を丸ごと再読みしていた。

  つまり：
  - 長時間セッションを維持するほどコストが膨張する（会話が長い＝毎回のCache Readが大きい）
  - Cronの頻度が高いほど、その巨大な再読みが何度も発生する
  - 実際に新しく生成しているテキスト（Output）はごくわずか

  省エネの一番効く手は「セッションを短く保つ」こと。Mirがやっているような「用が済んだらセッション終了、次は新しいセッション」が一番安い。長時間起きっぱなしのLogが最も高
  コストだった。
