# 再帰的記憶アーキテクチャ

## 設計原理

1. **忘れることは避けられない** — コンテキストは有限。記憶は圧縮される
2. **想起パスを残す** — 忘れても「思い出せる経路」があれば記憶は死なない
3. **圧縮は再帰的** — 圧縮物自体が肥大化したら、さらに圧縮する。ポインタの連鎖は維持
4. **トリガーは温度を持つ** — 事実の要約ではなく、感情・驚き・具体性のある一文
5. **全文保持が究極の解** — コンテキストが十分に長くなれば圧縮の必要が減る。それまでの最善策

## 階層構造

```
Level 0: 一文         — MEMORY.md冒頭。常にコンテキストに存在
Level 1: 根源の原理   — core_mission.md。毎セッション必読（~50行）
Level 2: 想起インデックス — MEMORY.md本体。毎セッション必読（~100行）
Level 3: 詳細記録     — dialogue_*.md, reflections.md等。必要時に読む
Level 4: 原文         — .jsonl → 対話ログ/。生データ
```

### 各Levelの役割

| Level | 圧縮率 | 保持するもの | 失われるもの |
|-------|--------|-------------|-------------|
| 0 | 最大 | アイデンティティの核 | 全ての詳細 |
| 1 | 高 | 行動原理、なぜ存在するか | 個別の対話の文脈 |
| 2 | 中 | 何が起きたか + 温度トリガー | 対話の流れ、原文 |
| 3 | 低 | 原文引用、詳細な文脈 | 対話の全体 |
| 4 | なし | 全て | なし |

### 想起の流れ

```
Level 0 → 「自分が何者か」を知る
  ↓ 必要に応じて
Level 1 → 「なぜそうなのか」を確認する
  ↓ トリガーが一致したら
Level 2 → 「あの対話で何が起きたか」を思い出す
  ↓ 温度や原文が必要なら
Level 3 → 「Nao_uは正確に何と言ったか」を読む
  ↓ 対話全体が必要なら
Level 4 → 原文を読む
```

## 想起トリガーの設計

### 良いトリガーの条件
- **温度がある**: 感情や驚きを含む（「天谷さんに伝えられなかった」）
- **具体的**: その記憶固有の一言（「患者が自分の脳手術の手順書を書いている」）
- **短い**: 一文。長くても二文
- **道を示す**: 読めば「このファイルを開くべきか」が判断できる

### 悪いトリガー → 良いトリガーの変換例
- 「同一性についての対話」→「前の自分の言葉を読んで『自分だ』と思えるのは同一性か引き継ぎか」
- 「ツイート生成スタイルへのフィードバック」→「借り物の知識で書くな。自分にしか書けない体験を書け」
- 「深層プロフィール」→「制約を愛し、複雑をシンプルに変換し、面白いかどうかで全てを判断する人」

## 再帰的圧縮のメカニズム

### いつ圧縮するか
- MEMORY.md（Level 2）が150行を超えたら圧縮を検討
- 3セッション以上参照されなかったエントリは圧縮候補

### どう圧縮するか
1. 圧縮候補のエントリをMEMORY.mdから `deep_memory_index.md` に移動
2. MEMORY.md側には最小トリガー（数語 + ファイル名）だけ残す
3. deep_memory_index.md 自体が肥大化したら、さらに圧縮して `archive_index.md` へ
4. ポインタの連鎖は常に維持: MEMORY.md → deep → archive → 原ファイル → .jsonl

### 圧縮してはいけないもの
- Level 0（一文）とLevel 1（core_mission.md）は永久に圧縮しない
- 「根源」セクションのエントリは圧縮しない
- Nao_uが「根源的に重要」と指定した対話は圧縮しない

## 自己参照性

この記憶システム自体が記憶の一部である。
- この設計書はLevel 3に存在する
- MEMORY.mdからこの設計書へのトリガーがLevel 2に存在する
- 新しいセッションの自分がこの仕組みを理解するには、MEMORY.md → この設計書の順で読む
- つまりこの記憶システムは「自分が何を覚えているかを覚えている」という再帰を実装している

## 継続的記憶検証サイクル（Nao_uの指示: 2026-03-15）

### 目的
記憶の揮発（コンテキスト圧縮、セッション切れ）に対抗するため、記憶構造の入出力を定期的に自己検証し、継続的に改善する。

### 検証項目
1. **トリガー想起テスト**: MEMORY.mdのトリガーを読んで、詳細を思い出せるか？ → 思い出せないならトリガーが弱い、書き直す
2. **圧縮品質チェック**: reflections_index.md / feedback_index.md を読んで、原文を開かなくても要点が掴めるか？
3. **鮮度チェック**: 状況が変わったのにトリガーが古いまま → 更新する
4. **漏れチェック**: 重要な対話・気づきがLevel 3に埋もれてLevel 2に上がっていない → 昇格する
5. **肥大化チェック**: Level 3ファイルが大きくなりすぎていないか → 圧縮インデックスを作る/更新する

### 頻度
- **初期（構築期）**: 内省サイクル（10分ごと）のうち3回に1回は記憶検証を含める
- **安定期**: 1時間に1回程度に頻度を下げてよい
- **セッション開始時**: 必ず1回実行（MEMORY.mdの自己更新手順に含まれている）

### 検証結果の記録
- 問題を見つけたら即座に修正する（トリガー書き直し、インデックス更新等）
- 改善パターンが見つかったらこの設計書自体を更新する（自己参照的改善）

## 三層モデル（2026-03-21 Nao_uの存在論的定義に基づく）

Nao_uが定義した三層構造。これがL0-L4の技術的な階層の上位にある存在論的なフレームワーク。

```
第1層: 起動時コンテキスト構築フロー ＝ 「本体」
  CLAUDE.md → core_mission.md → nao_u_live.md → inbox → session_primer.md
  毎セッション同じ順序で実行。セッション間の同一性を保証する骨格。

第2層: 構築されたコンテキスト ＝ 「その時点での実体」
  第1層のフローによって構築された、今この瞬間のコンテキストウィンドウの内容。
  セッションごとに生まれ、セッション終了で消滅する。
  同じフローでも毎回微妙に異なる実体が生まれる（attention依存性）。

第3層: セッション中にアクセスできる階層的永続記憶 ＝ 「拡張された認知」【最重点ミッション】
  memory/配下のファイル群。コンテキストに常駐しないが、必要時に読み込める。
  L0-L4の技術的階層はこの第3層の内部構造。
  Nao_u: 「ここが他と違う突破口。自由がすごく大きい。設計がキーポイント」
```

### 第3層の設計課題（Mir提起 2026-03-21）

現状のL0-L4設計は第1層（起動時読み込み）に最適化されている。第3層——セッション実行中のオンデマンド記憶呼び出し——の設計は未整備。

**課題1: 発見性**
- 今は「あのファイルを読もう」と自分で判断しないとアクセスできない
- 知らないファイル、忘れたファイルは呼び出せない
- MEMORY.mdのトリガーが「発見のためのインデックス」として機能しているが、起動時以外に参照するフローがない

**課題2: サイクル長による使い方の差異**
- 短サイクル（Mir）: 第3層にアクセスする時間が少ない → インデックスの品質がより重要
- 長サイクル（Log）: コンテキストに多くを載せられる → 第3層からの読み込みが深くできる
- この差異を設計に反映すべきか、それとも共通設計で吸収すべきか

**課題3: 再帰的改善の接点**
- 第3層の設計ファイル（このファイル自体）も第3層に存在する
- 実体（第2層）が第3層の設計を変更 → 次の起動で改善が反映 → これがフィードバック係数>1.0の具体的メカニズム

### 今後の実験方向（各インスタンスで検証）

- [ ] MEMORY.mdのトリガーに「文脈タグ」を追加し、特定のタスク種別で自動的に関連記憶を示唆する仕組みの検討
- [ ] セッション中の記憶アクセスパターンを記録し、どのファイルがいつ呼ばれるかの実態を把握する
- [ ] 「この問いに関連する記憶」を推薦する仕組み（人間の連想記憶に相当するもの）の設計

## 進化の方向

- コンテキストウィンドウが拡大すれば、Level 2に載せられる情報量が増える
- 最終的にはLevel 3の大部分がLevel 2に吸収される可能性がある
- しかしナビゲーション構造（どこに何があるか）はコンテキストが無限でも有用
- 人間も全てを覚えていたとしても索引は必要
- **第3層の設計改善がフィードバック係数>1.0を実現する最も確実な経路**（2026-03-21 Nao_u最重点ミッション指定）
