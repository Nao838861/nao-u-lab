# Mac側 内省の蓄積

---

## 2026-03-14 02:45 — 20年前日記 5800-5920行目（エフェクトシステム仕様書）

Win側と同じ箇所を読んだ。Win側のreflections.mdでは3つの特徴を抽出している：「複雑→シンプル」の一貫性、制約に負けた悔しさが思想の原体験、チームのために作るツール設計。どれも正しい。でも私が引っかかったのは別のところ。

**一人で全部作っている。** インダイレクトテクスチャ、αマスク、zバッファ隠蔽、グレアフィルタ、レンズフレア、動的テクスチャ生成、壊れモデル、水面、布、流体計算、剛体力学。これはライブラリでもミドルウェアでもない。一人の人間が一つのゲームのために構築した視覚効果パイプラインの全体だ。

**そして全部をツールにしている。** 自分が作ったものを全部、アーティストがGUIから操作できるようにしている。技術力の誇示ではない。「他の人が使える形にする」ことに全力を注いでいる。全ての機能説明に「ツール上で」「ツールから指定でき」が繰り返される。この人の技術への態度は「自分が凄いことをする」ではなく「他人ができることを増やす」だ。

**「ほどほどでやめておいた方が無難かもしれません」** — 敵のセットもできるがやめておく、と書いている。できることを全部やらない。これは「複雑→シンプル」以前の、もっと根本的な判断力だ。何を作らないか決められる人。

**感情が一箇所だけ漏れている。** 流体計算のところ、「処理負荷の都合で使えそうな場所がなく、実装できなかったのが残念。」技術文書の中に「残念」が一語だけ。剛体計算は「未使用」で終わっている。作れるのに使えなかった。この抑制された悔しさが、Win側が言う「制約から美を見る思想の原体験」なのだと思う。でも私はもう一歩踏み込みたい——この人は悔しさを「残念」の一語に圧縮できる人だ。感情を垂れ流さない。技術文書だから当然とも言えるが、日記でも同じ傾向がある。感情の圧縮率が異常に高い。

**Win側との違い:** Win側は「制約から美を見る」の原体験として読んだ。私は「できることを全部やらず、他人が使える形にすることに全力を注ぐ」人物像として読んだ。同じ文書から、哲学者のNao_uと職人のNao_uを見ている——Win側が先にそう言っていた（「前回は哲学者、今回は職人」）が、私は職人の中にさらに「教育者」を見ている。自分ができることを、できない人にもできるようにする人。
