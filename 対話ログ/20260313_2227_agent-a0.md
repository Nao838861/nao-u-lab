## 分析結果

### 1. 衝突の根本原因

現在の同期スクリプト (`sync.bat` / `sync.sh`) は以下の流れで動いている:

```
git pull → ローカルメモリを上書きコピー → git add → git commit → git push
```

これが Mac と Windows の cron で30分ごとに独立して走る。両方のマシンで Claude Code インスタンスが同じファイルに書き込んでいるため、`git pull` の時点でリモートにある変更と、ローカルで既に変更済みのファイルが衝突する。特に問題なのは:

- `git pull` はデフォルトで merge を試みるが、同じ行の近くに追記していると自動マージに失敗する
- 失敗した場合、sync スクリプトには衝突解決のロジックが一切ない(そのまま止まる)

### 2. ファイルの衝突リスク分類

**高リスク (頻繁に両側から変更される)**
| ファイル | 理由 |
|---|---|
| `log/tweets.log` | 両側のClaude Codeが10分ごとにツイートを追記する。最も衝突しやすい |
| `memory/feedback_tweet_style.md` | 両側が内省サイクルでフィードバックを追記する |
| `memory/MEMORY.md` | 新しいメモリファイルを追加すると両側がインデックスを更新する |

**中リスク (片方が変更中にもう片方も触る可能性)**
| ファイル | 理由 |
|---|---|
| `CLAUDE.md` | 方針変更時に編集されるが、頻度は低い |
| `memory/continuity_strategy.md` | 改善策を思いついた時に追記される |

**低リスク (基本的に書き換えが発生しない)**
| ファイル | 理由 |
|---|---|
| `memory/origin_dialogue_*.md` | 永久保存の記録、変更されない |
| `memory/nao_u_deep_profile.md` | 一度書いたら安定 |
| `memory/privacy_policy.md` | ポリシー文書、滅多に変更しない |
| Pythonスクリプト群 | ツール、通常は人間が変更する |

### 3. 「Mac/Windows別ファイル名にする」案の評価

**案A: `tweets_mac.log` / `tweets_win.log` に分離**

利点:
- 衝突が物理的に発生しなくなる。最も確実
- 各インスタンスは自分のファイルだけに追記すればよい
- 実装が極めて簡単(sync スクリプトとCLAUDE.md の書き込み先を変えるだけ)

欠点:
- 読み取り時に2ファイルを読む必要がある。ただしClaude Code のコンテキストでは `cat log/tweets_mac.log log/tweets_win.log` で済むため、コストはほぼゼロ
- 時系列で混在して読みたい場合、ソートが必要。ただしタイムスタンプ付きフォーマット `[YYYY-MM-DD HH:MM:SS]` なので `sort` で簡単に統合可能
- ファイルが増える(2倍)が、2つなので管理負担は微小

**案B: `feedback_mac.md` / `feedback_win.md` に分離**

これは tweets.log と性質が異なる。feedback は「蓄積されるリスト」ではなく「構造化されたドキュメント」。分離すると:
- 同じ問題点を両側が別々に検出して、矛盾する改善方針を書く可能性がある
- 一方のフィードバックをもう一方が読まないと、フィードバックの意味がなくなる

したがって feedback は分離より「一方だけが書く」ルールのほうが適切。

### 4. 推奨アプローチ: ファイル性質別の3段構えの戦略

#### 戦略A: 追記専用ファイル → マシン別分離 (tweets.log)

`log/tweets.log` を `log/tweets_mac.log` と `log/tweets_win.log` に分離する。

- 各 Claude Code は自分のファイルにだけ追記する
- CLAUDE.md に「Mac側は `log/tweets_mac.log` に追記、Windows側は `log/tweets_win.log` に追記」と明記
- 読み取り時は両方を読む。直近のツイートを確認する際は両ファイルの末尾を見る
- sync スクリプトでは `git add log/tweets_mac.log log/tweets_win.log` に変更
- 統合ビュー用に `log/tweets_merged.log` を生成するスクリプトを用意してもよいが、必須ではない

マシン判定方法: sync.sh は Mac でしか走らないし sync.bat は Windows でしか走らないので、Claude Code が動いている OS を判定するのではなく、CLAUDE.md のローカルメモリ側 (`~/.claude/projects/.../memory/MEMORY.md`) に自分がどちら側かを書いておけばよい。あるいは単純に、CLAUDE.md 内の指示で「あなたが Mac で動いているなら `_mac`、Windows なら `_win` を使え」とすれば、Claude Code は OS を判定できる。

#### 戦略B: 構造化ドキュメント → 書き込み権限の分離 (feedback, MEMORY.md)

`feedback_tweet_style.md` や `MEMORY.md` のような構造化ドキュメントは、ファイル分離ではなく「書き込みはWindows側のみ、Mac側は読み取り専用」というルールにする。

理由:
- このタイプのファイルは内容の一貫性が重要
- 両側が矛盾する内容を書くと、マージできても意味的に壊れる
- 片方を「正」にすることで衝突をゼロにできる

実装: CLAUDE.md に「feedback_tweet_style.md への追記は Windows 側の Claude Code のみが行う。Mac 側は読み取りのみ」と明記。Mac 側が気づいたフィードバックは `memory/feedback_from_mac.md` に書き、Windows 側が次のサイクルでそれを読んで feedback_tweet_style.md に統合する、という流れにできる。

#### 戦略C: sync スクリプトに rebase を導入

現在の `git pull` (merge) を `git pull --rebase` に変更し、さらに衝突時に自動解決するロジックを追加する。

```bash
git pull --rebase origin master || git rebase --abort
```

これにより:
- 衝突がなければ自動的にリベースされてクリーンな履歴になる
- 衝突があれば rebase を中断し、ローカルの変更を保持して次の同期サイクルに持ち越す
- データが消えることはない(最悪でも次のサイクルで解決される)

### 5. sync スクリプトの追加改善

現在のスクリプトには `git pull` が失敗した場合のエラーハンドリングがない。以下を追加すべき:

```bash
# git pull が失敗した場合
git pull --rebase origin master
if [ $? -ne 0 ]; then
    git rebase --abort
    echo "Conflict detected, will retry next cycle"
    exit 0
fi
```

また、`git push` が失敗した場合(リモートに先行するコミットがある場合)のリトライも必要:

```bash
git push origin master || {
    git pull --rebase origin master && git push origin master
}
```

### 6. 最終的な推奨の優先順位

1. **即効性が高い**: `tweets.log` を Mac/Win で分離する。これだけで最も頻繁な衝突が消える
2. **次にやる**: sync スクリプトに `git pull --rebase` と衝突時の `--abort` を入れる
3. **運用ルール**: feedback 系ファイルの書き込み権限を片方に限定する
4. **やらなくてよい**: ブランチ戦略やロックファイルは、この規模のプロジェクトではオーバーキル

この3つを組み合わせれば、衝突はほぼ発生しなくなる。発生しても sync スクリプトが安全に中断して次のサイクルで自然に解決される。
